# Security Audit Report — 2026-06-01

## Methodology
Source code review of all frontend files, Edge Functions, migrations, and build scripts. Verified each claim against actual code in the repository. No npm audit was used for these findings — these are architectural and code-level issues.

---

## Findings

### 1. CRITICAL — RLS Policy Regression on Email Tables
**Status:** CONFIRMED
**File:** `supabase/migrations/20250723000003_update_rls_policies.sql`

Policies were downgraded from admin-only to any-authenticated-user:
- Before: `raw_user_meta_data->>'role' = 'admin'`
- After: `auth.role() = 'authenticated'`

No later migration reverts this. Any user who can log into Supabase Auth can read, modify, and delete email queue data and analytics.

**Risk:** Data tampering, email injection, analytics manipulation
**Fix:** Create a new migration that restores admin-only RLS policies.

### 3. HIGH — Consent Cookies Missing Secure Flag
**Status:** CONFIRMED
**File:** `src/lib/consent.ts` (line 28)

```typescript
document.cookie = `${name}=${value}; expires=${daysToExpires(days)}; path=/; SameSite=Lax`;
```

Missing `; Secure`. Cookies will be transmitted over HTTP, which is a GDPR compliance concern.

**Risk:** Cookie theft on MITM networks; GDPR non-compliance
**Fix:** Add `; Secure` to the cookie string.

### 4. HIGH — Mermaid SVG Rendered via dangerouslySetInnerHTML
**Status:** CONFIRMED
**File:** `src/components/markdown/components/MermaidBlock.tsx` (line 94)

Mermaid renders diagrams to SVG and injects them via `dangerouslySetInnerHTML`. While `securityLevel: 'strict'` is set, the SVG output is not sanitized after rendering. Attacker-controlled blog content with malicious Mermaid code could produce SVG containing `<a href="javascript:...">` or event handler attributes.

DOMPurify (v3.4.7) exists as a transitive dependency of Mermaid but is not used directly by the application.

**Risk:** XSS via attacker-controlled blog content
**Fix:** Sanitize SVG output through DOMPurify before rendering.

---

### 5. MEDIUM — Email Preview HTML Rendered Client-Side
**Status:** CONFIRMED
**File:** `src/pages/EmailDashboard.tsx` (line 929)

Preview panel renders raw HTML from the `render-email-preview` Edge Function via `dangerouslySetInnerHTML`. The HTML is generated server-side from templates that interpolate Supabase data (post titles, excerpts, tags). If any database content contains unescaped HTML, it executes in the admin panel.

**Risk:** XSS in admin panel via crafted database content
**Fix:** Sanitize HTML on the Edge Function side before returning, or use DOMPurify client-side.

### 6. MEDIUM — rehype-raw Bypasses Sanitization Ordering
**Status:** CONFIRMED
**File:** `src/components/markdown/GfmMarkdown.tsx` (lines 470-472)

```typescript
const plugins: PluggableList = [rehypeRaw];
if (mergedConfig.sanitizeHtml) {
  plugins.push([rehypeSanitize, mathSchemaExtensions]);
}
```

`rehype-raw` always runs before `rehype-sanitize`. Raw HTML enters the AST as full DOM before sanitization strips dangerous elements. The sanitize schema extends the default to allow video/audio/track but doesn't explicitly block iframe/object/embed (those are already blocked by defaultSchema).

**Risk:** Potential XSS if new tag types slip through sanitize schema
**Fix:** Consider removing `rehype-raw` if not needed, or add explicit tagNames blocks.

### 7. MEDIUM — Unpinned CDN Imports in Edge Functions
**Status:** CONFIRMED
**Files:** All Edge Functions in `supabase/functions/`

Inconsistent import versions:
- `https://esm.sh/@supabase/supabase-js@2` — unpinned major version (used by geolocation, send-newsletter-emails)
- `https://esm.sh/@supabase/supabase-js@2.7.1` — pinned (used by render-email-preview)
- `https://deno.land/std@0.168.0/http/server.ts` — pinned but old (used by most)
- `https://deno.land/std@0.177.0/http/server.ts` — different version (used by spotify-now-playing)

**Risk:** Supply chain attack if CDN compromised; unexpected breaking changes
**Fix:** Pin all imports to exact versions. Consider vendoring.

### 8. MEDIUM — ip-api.com Called Over HTTP
**Status:** CONFIRMED
**File:** `supabase/functions/geolocation/index.ts` (line 48)

```typescript
const geoResponse = await fetch(`http://ip-api.com/json/${ip}?fields=...`)
```

The geolocation API is called over unencrypted HTTP. Exposes user IP to interception and allows MITM on the response.

**Risk:** IP address exposure; response tampering
**Fix:** Change to `https://`. ip-api.com supports HTTPS.

---

### 9. MEDIUM — Overly Permissive CORS on All Edge Functions
**Status:** CONFIRMED
**Files:** All 7 Edge Functions use `'Access-Control-Allow-Origin': '*'`

All functions allow any origin:
- email-webhook, geolocation, handle-unsubscribe, render-email-preview, send-newsletter-emails, strava-proxy
- spotify-now-playing uses dynamic origin (`req.headers.get("origin") ?? "*"`)

For public-facing functions (webhook, unsubscribe) this is acceptable. For data-querying functions (render-email-preview, geolocation), wildcard CORS increases the attack surface.

**Risk:** Data exfiltration vector via cross-origin requests
**Fix:** Restrict to `https://imadlab.com` on functions that query Supabase data.

---

### 10. LOW — No Content-Security-Policy Header
**Status:** CONFIRMED

No CSP headers found in source code, Vite config, or build scripts. The deployed site has no protection against XSS, clickjacking, or content injection.

**Risk:** Increased impact of any XSS vulnerability
**Fix:** Add CSP via Cloudflare Edge Workers or hosting provider middleware.

---

### 11. LOW — Strava Proxy Has No Auth Validation or Rate Limiting
**Status:** CONFIRMED
**File:** `supabase/functions/strava-proxy/index.ts` (122 lines, no auth check)

The function does not validate Supabase authentication from the caller. It only checks for its own stored credentials (STRAVA_CLIENT_ID, etc.). Any authenticated Supabase user can invoke it to make Strava API calls.

No rate limiting exists in the function itself (Strava's own limits apply, but the proxy doesn't enforce per-user limits).

**Risk:** Abuse of Strava API quota by authenticated users
**Fix:** Add Supabase auth validation and per-user rate limiting.

---

### 12. LOW — Hardcoded IndexNow Fallback Key
**Status:** CONFIRMED
**File:** `scripts/submit-indexnow.js` (line 5)

```javascript
const DEFAULT_INDEXNOW_KEY = '47bf4593ec4fc33700cda5cecf0d7fec';
```

The script checks if the key file exists before using it, so risk is minimal. But a hardcoded fallback defeats the purpose of per-deployment key generation.

**Risk:** Low — only matters if public/key file is missing
**Fix:** Remove fallback. Fail loudly if no key is configured.
